# Bug Fix Log — 2026-05-27

## 症状

客户端快速重启后，TCP 连接能建立（三次握手成功），但服务端收不到任何数据，连接建立后无响应。

## 根因分析

两个 bug 叠加导致连接在 epoll 中静默失效：

### Bug 1: `del_event` 从未被调用 — `ev->active` 残留

`rps_close_connection()` / `rps_free_connection()` 只做 `close(fd)` + 销毁 pool，从不调用 `del_event` 将 fd 从 epoll 中移除。内核在 `close(fd)` 时自动从 epoll 踢掉 fd，但 `ev->active` 和 `ev->epoll_events` 保持旧值（`active = 1`）。

当该连接被回收后再次分配时（`rps_event_accept` → `rps_get_connection`），`rps_epoll_add_event` 看到 `ev->active == 1`，走了 `EPOLL_CTL_MOD` 分支而非 `EPOLL_CTL_ADD`。

**文件:** `src/core/rps_connection.c:89-97`

### Bug 2: `EPOLLEXCLUSIVE` + `EPOLL_CTL_MOD` 非法

`rps_epoll_add_event` 对所有事件都设置了 `EPOLLEXCLUSIVE`。Linux `epoll_ctl(2)` 明确禁止 `EPOLLEXCLUSIVE` 与 `EPOLL_CTL_MOD` 同时使用：

> EPOLLEXCLUSIVE may be specified only with the EPOLL_CTL_ADD operation.

两个 bug 叠加链：
1. 连接回收 → `ev->active = 1`（Bug 1 残留）
2. 连接复用 → `EPOLL_CTL_MOD` + `EPOLLEXCLUSIVE` → `epoll_ctl` 失败
3. `add_event` 返回值未被检查 → 静默失败
4. 新 fd 不在 epoll 中 → 永远收不到读事件

**文件:** `src/event/modules/rps_epoll_module.c:119,129-134`

## 修复内容

### 1. `src/core/rps_connection.c` — `rps_close_connection` 增加事件清理

在 `close(fd)` 之前调用 `del_event` 分别清理读写事件，确保 `ev->active = 0`，下次 `add_event` 走 `EPOLL_CTL_ADD`。

### 2. `src/event/modules/rps_epoll_module.c` — 移除 `EPOLLEXCLUSIVE`

将所有事件的 `ee.events = EPOLLEXCLUSIVE` 改为 `ee.events = 0`。

`EPOLLEXCLUSIVE` 仅对多进程/多线程 accept 同一 listen socket 有意义（防止惊群），当前单 worker 架构不需要。后续如需可单独对 listen fd 的事件加回。

### 3. `src/core/rps.c` — 检查 `add_event` 返回值

- `rps_event_accept`（新连接注册读事件）：失败时 `rps_free_connection` + 打错误日志
- `done:` 标签 keepalive 路径（重建 request 后重新注册）：失败时 `rps_free_connection` + 打错误日志

### 4. `build/rps_test/rps_echo_client.c` — 测试工具优化

- 连接保持持久，不在每次发送后 `shutdown()` 或关闭
- 支持拆包模拟：`-c N` 控制每 chunk 字节数（默认 1），`-d N` 控制 chunk 间延迟（微秒）
- stdin 输入支持转义序列：`\r` `\n` `\t` `\\` `\0`
- 移除自定义 SIGINT 处理，依赖内核默认行为，`read()` 阻塞时 Ctrl+C 也能立即退出
- 添加 `SO_REUSEADDR`
