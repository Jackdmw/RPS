# 2026-06-12 工作日志

## 一、Proxy 请求构造优化

### 消除临时缓冲 + memcpy
`rps_http_proxy_build_request` 合并入 `rps_http_proxy_create_request`，之前流程：
1. `rps_palloc(pool, 8192)` 分配临时 raw buffer
2. 构造 HTTP 请求到临时 buffer
3. `rps_buf_create(r->pool, len)` 创建 `u->request_bufs`
4. `memcpy` 从临时 buffer 拷贝

现在直接写入 `u->request_bufs->pos`，写完设 `last`，一次分配零拷贝。

### Header 去重（hash 加速）
- `rps_string.h` 新增 `rps_hash_str_lc(str, hash)`（大小写不敏感 djb2）
- `rps_http_proxy_header_t` 新增 `key_hash` 字段，配置解析时预计算
- 转发客户端 header 时：内置头（host/connection/x-forwarded-for）直接跳过；`proxy_set_header` 覆盖的头通过 hash 先行 + `rps_strcmp` 确认来去重

### X-Real-IP / X-Forwarded-For 策略修正
- **X-Real-IP**：客户端已设置则保留（多层代理透传原始 IP），否则写入直连 IP
- **X-Forwarded-For**：追加 `, <current_ip>` 而非覆盖

### 后端 Connection 头策略
- 请求构造时：`upstream_conf->keepalive > 0` 发 `Connection: keep-alive`，否则 `close`
- 响应解析时：`process_header` 根据后端 HTTP 版本 + Connection 响应头设 `u->keepalive`

## 三、upstream 协议回调抽象

### forward_body 回调
`rps_upstream_t` 新增 `rps_int_t (*forward_body)(r, u)`，body 转发解耦：

| 协议 | forward_body 实现 |
|------|-------------------|
| proxy | `rps_http_proxy_forward_body` — 走 HTTP 过滤链 (`rps_http_send_body` → `write_filter`) |
| WebSocket (预留) | 直接透传 fd |

body 转发链：read_handler 读后端数据 → forward_body → HTTP 过滤链 → 客户端。客户端写阻塞时背压（del 后端读事件 + body_write_continue）。

## 四、致命 Bug 修复

### 1. Use-After-Free：`r->connection` 在 `finalize_request` 后访问
`rps_http_finalize_request` → `rps_http_close_request` → `rps_destroy_pool(r->pool)` 释放了 r。
之后 `rps_http_complete_request(r->connection)` 访问已释放内存。

**修复**：新增 `rps_http_finalize_and_complete(r, rc)` 辅助函数，在 `finalize` 前保存 `c = r->connection`，替换 phases.c 中 10 处 + upstream.c 中 1 处调用。

### 2. keepalive 后端不发 FIN → request 永不释放
后端 keep-alive 时响应完成后不关连接，`read_handler` 永远等不到 `n==0`，request 一直占着。

**修复**：
- 新增 `content_length_n` / `body_received` 追踪字段
- proxy `process_header` 解析后端 Content-Length 存入 `u`
- `read_handler`：body 收全 + 转发完 → 直接 `finalize`，不依赖后端 EOF
- `body_write_continue`：写完成后同样检查

### 3. 死连接被缓存
后端 idle timeout 关闭连接 → `n==0` → `finalize` → `free_peer` 检查 `u->keepalive`（后端响应头说 keep-alive，还是 1）→ 把已 FIN 的连接缓存。下个请求取到死连接 → `send()` 失败 → 502。

**修复**：`read_handler` 中 `n==0` 时设 `u->keepalive = 0`，`free_peer` 走 close 路径不入缓存。

### 4. LT epoll busy-poll
请求进入阶段引擎后 `c->data = NULL`。客户端在请求完成前发新数据 → epoll (LT) 反复触发 handler → handler 发现 `r==NULL` 直接 return → 数据未读走 → epoll 再次触发 → busy loop。

**修复**：`rps_http_wait_request_handler` 中 `r==NULL` 时 `del_event(c->read)`，等 `rps_http_complete_request` 创建新 request 后 `add_event` 重新注册。加保护：write 事件活跃时不 del（避免误删正在写响应的 fd）。

### 5. 客户端读超时未重置
keepalive 连接复用时，计时器从上一个请求继承，不是新鲜 60s。

**修复**：`rps_http_create_request` 中统一调用 `rps_event_add_timer(c->read, 60000)` + `request->start_msec = rps_current_msec()`。`complete_request` 和 `rps_event_accept` 中的重复 `add_timer` 移除。
